## 🎯 Overview

### Why Secrets Are Needed

GitHub Actions workflows need access to:
- **Firebase deployment token** - Deploy hosting, rules, and functions
- **Firebase client configuration** - Build Vite apps with Firebase SDK
- **Observability credentials** (optional) - Push metrics to monitoring services

### Secrets vs Variables

| Type | Use Case | Visibility | Example |
|------|----------|------------|---------|
| **Secrets** | Sensitive credentials | Masked in logs | `FIREBASE_TOKEN` |
| **Variables** | Non-sensitive config | Plain text in logs | `VITE_FIREBASE_PROJECT_ID` |

**Firebase client keys** can be **variables** (not secrets) because:
- They're meant for public client apps
- Access control is via Firebase Security Rules, not API key secrecy
- They appear in client bundles anyway

## 🔑 Required Secrets

### 1. FIREBASE_TOKEN

**Type:** Secret (masked)  
**Purpose:** Deploy to Firebase Hosting, Realtime Database, Firestore, Functions  
**Used in:**
- `.github/workflows/ci.yml` - Main pipeline
- `.github/workflows/deploy-and-test.yml` - Manual deploy
- `.github/workflows/deploy-flutter.yml` - Flutter deploy
- `.github/workflows/deploy-with-labels.yml` - Label-based deploy

**How to get:**

```bash
# Install Firebase CLI (if not installed)
npm install -g firebase-tools

# Login and generate CI token
firebase login:ci

# Output example:
# ✔ Success! Use this token to login on a CI server:
# 1//0gABCdefGHIjklMNOpqrSTUvwxyz...
```

**Add to GitHub:**

```bash
# Using GitHub CLI
gh secret set FIREBASE_TOKEN
# Paste token when prompted

# Or via GitHub UI
# Settings → Secrets and variables → Actions → New repository secret
# Name: FIREBASE_TOKEN
# Secret: (paste token)
```

## 📝 Step-by-Step Setup

### Option 1: GitHub Web UI

**For Secrets:**

1. Go to your repository on GitHub
2. Click **Settings** tab
3. Navigate to **Secrets and variables** → **Actions**
4. Click **New repository secret**
5. Enter:
   - **Name:** `FIREBASE_TOKEN`
   - **Secret:** (paste your firebase login:ci token)
6. Click **Add secret**

**For Variables:**

1. In the same **Secrets and variables** → **Actions** page
2. Click the **Variables** tab
3. Click **New repository variable**
4. Enter:
   - **Name:** `VITE_FIREBASE_API_KEY`
   - **Value:** (paste from Firebase Console)
5. Click **Add variable**
6. Repeat for all 9 Firebase variables

---

### Option 2: GitHub CLI

**Install GitHub CLI:**

```bash
# macOS
brew install gh

# Linux
# See: https://github.com/cli/cli/blob/trunk/docs/install_linux.md

# Windows
# See: https://github.com/cli/cli#installation
```

**Authenticate:**

```bash
gh auth login
```

**Set secrets:**

```bash
# Set FIREBASE_TOKEN
gh secret set FIREBASE_TOKEN
# Paste token when prompted, press Enter

# Set Firebase variables (as secrets - they'll be masked)
gh secret set VITE_FIREBASE_API_KEY
gh secret set VITE_FIREBASE_AUTH_DOMAIN
gh secret set VITE_FIREBASE_DATABASE_URL
gh secret set VITE_FIREBASE_PROJECT_ID
gh secret set VITE_FIREBASE_STORAGE_BUCKET
gh secret set VITE_FIREBASE_MESSAGING_SENDER_ID
gh secret set VITE_FIREBASE_APP_ID
gh secret set VITE_FIREBASE_MEASUREMENT_ID
```

**Set as variables (visible in logs):**

```bash
# GitHub CLI doesn't support variables directly yet
# Use web UI: Settings → Secrets and variables → Actions → Variables tab
```

### Option 3: Bulk Import from .env

**Create a temporary file** (don't commit!):

```bash
# secrets.env
FIREBASE_TOKEN=1//0gABCdefGHIjklMNOpqrSTUvwxyz...
VITE_FIREBASE_API_KEY=AIzaSyABCDEFGHIJKLMNOPQRSTUVWXYZ
VITE_FIREBASE_AUTH_DOMAIN=shiritori-game-ccaae.firebaseapp.com
# ... etc
```

**Import using GitHub CLI:**

```bash
gh secret set -f secrets.env

# Then immediately delete
rm secrets.env
```

## 🐛 Troubleshooting

### "Missing Firebase environment variables"

**Error in build logs:**
```
Error: Missing Firebase environment variables: VITE_FIREBASE_API_KEY, VITE_FIREBASE_APP_ID.
Copy .env.example to .env and set values from Firebase Console.
```

**Solution:**
1. Check secrets are set: `gh secret list`
2. Verify workflow has `env:` block passing secrets
3. Check `.github/workflows/ci.yml` lines 333-341 (shiritori-online build)
4. Check `.github/workflows/ci.yml` lines 412-422 (kawaii-shiritori build)

**Fix:**
```bash
gh secret set VITE_FIREBASE_API_KEY
gh secret set VITE_FIREBASE_APP_ID
# ... set all required variables
```

---

### "FIREBASE_TOKEN not set"

**Error in deploy jobs:**
```
Error: FIREBASE_TOKEN not set
```

**Solution:**
```bash
# Generate new token
firebase login:ci

# Set secret
gh secret set FIREBASE_TOKEN
```

**Check token works:**
```bash
firebase projects:list --token "$FIREBASE_TOKEN"
```

---

### "Deploy failed: HTTP 401 Unauthorized"

**Cause:** Firebase token expired or invalid

**Solution:**
```bash
# Logout and re-login
firebase logout
firebase login:ci

# Update secret
gh secret set FIREBASE_TOKEN
```

---

### Build succeeds but deploy skipped

**Check workflow conditions:**

1. Deploy jobs only run on `push` events, not `pull_request`
2. Deploy requires specific branches (main, develop)
3. Check job `if:` conditions in workflow files

**Trigger manual deploy:**
```bash
# Using workflow dispatch
gh workflow run deploy-and-test.yml -f environment=staging

# Or push to main
git checkout main
git push origin main
```

---

### Secrets not available in forked repositories

**By design:** GitHub doesn't share secrets with forks (security measure)

**Workaround for contributors:**
1. Fork can't run deploy jobs (expected)
2. Builds will run if they don't depend on secrets
3. Maintainer runs deploy after merging PR

## 🔒 Security Best Practices

### ✅ Do

- **Use secrets for FIREBASE_TOKEN** - Deployment credential
- **Use variables for Firebase client config** - Public by design
- **Rotate FIREBASE_TOKEN regularly** - Every 90 days
- **Scope secrets to minimum workflows** - If possible
- **Test secrets in staging first** - Before production
- **Document all secrets** - In this file
- **Never commit .env files** - Use .env.example only

### ❌ Don't

- **Don't commit real .env files** - Only .env.example
- **Don't share FIREBASE_TOKEN in Slack/email** - Use secure channels
- **Don't use personal credentials** - Use project service accounts
- **Don't hardcode in source** - Always use environment variables
- **Don't expose in logs** - GitHub masks secrets automatically
- **Don't skip Firebase Security Rules** - Client keys need rule enforcement
